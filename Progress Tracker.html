2 Progress Tracker

The following description only plots little dots. You need to connect the dots by making conscious choices during your implementation. There will be points for the reason behind every choice made. Naturally, how well you connect the dots will determine how good the whole picture looks.

This project is to implement a classroom progress tracking system to help instructors analyse and make real time decisions.

Base Features:

  1. Instructor Interface
  1. A web interface with login support for (pre-)registered instructors.
  2. Forms for creating a Session for a particular class or tutorial with checkpoints and details under each checkpoint.
  3. Forms to populate a list of attending students along with a temporary password, either by entering each detail, or a CSV file.
  4. Activation, deactivation of a session and provision of an Active Session ID to be able to access the student interface.
  5. Monitoring an Active Session, with the following data displayed and refreshed every certain amount of time
  1. A histogram of number of students completed a particular checkpoint
  2. A histogram of number of pings across checkpoints
  3. A view of comments under each checkpoint
  4. A view of ping messages under each checkpoint
  1. Student Interface
  1. An Android application with provision to enter an Active Session with the Session ID.
  2. A login interface.
  3. Checkboxes for each checkpoint.
  4. Comment boxes under each checkpoint to convey questions/messages to the instructor.
  5. Ping feature under each checkpoint to send messages to the instructor to queue up doubts.
  1. Server
  1. The Server is to provide support for the Instructor Interface and the Student Application
  2. Instructor Interface
  1. Login and authentication based on entries in the database
  2. Collection of form data to create the session and populate it with details and storage and management in the database
  3. Storage of student ids and passwords in the database
  4. Maintenance of a database of active sessions and using it to accept/reject requests from the application.
  5. Servicing of all HTML requests for the web interface.
  1. Student Application
  1. Handling requests from the app for an active session
  2. Student authentication
  3. Handling of update requests corresponding to checkpoints for a particular student and updation in the database

Additional Features (Suggestions) [Min required = … ]

  1. Timeline mode
  1. After the conclusion of the session, the progress at a particular time during the session can be viewed and statistics of progress across time can be displayed to help instructors gain insight in planning future sessions.
  1. Broadcast the instructors screen so that students are able to follow the demonstration on their own personal screen instead of projector.
